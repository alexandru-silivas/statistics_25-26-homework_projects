Homework 7

Security Updates as a Biased Random Walk

We model a server that receives weekly security updates over n = 50 weeks. Every week, m independent attackers each try to breach the system, with probability p. If at least one attacker succeeds, the server is considered breached that week; otherwise, it's secure.

The probability the server remains secure for a week is s = (1 − p)m, and the breach probability is b = 1 − s. We translate weekly outcomes into a random walk: +1 for a secure week, −1 for a breached week.

Why This is a “Random Walk”

Interpreting +1 / −1 steps as a walk, the server’s cumulative score over time behaves like a biased random walk: the step bias depends on the number of attackers and their success probability. Over 50 weeks, different scenarios (e.g., few attackers vs many) lead to very different walk trajectories.

Fixed Demonstrative Simulations

Below are three fixed simulations with the same timescale (50 weeks) and fixed Y-axis (scores from −50 to +50):

  • Graph 1: One attacker for 50 weeks — the risk is relatively low, so the walk mostly stays near zero.
  • Graph 2: 50 attackers for 50 weeks — the breach probability is very high, so the walk tends to go down faster.
  • Graph 3: We run 500 independent trials (each trial is a 50-week walk with 50 attackers). This produces a “cloud” of random-walk paths, all drifting downward, showing how likely it is to accumulate a very negative score when breach risk is high.
